# GitHub 協作與版控指南

本文件說明團隊成員如何取得專案、在**自己的分支**開發，並透過 **Pull Request（PR）** 送審；由專案管理員審核通過後，才會合併進 `main`。

**儲存庫網址**：<https://github.com/Ohmomomo123/skin>

---

## 協作原則（必讀）

| 規則 | 說明 |
|------|------|
| **禁止直接改 `main`** | 協作者請勿 `git push` 到 `main` 分支 |
| **一律用功能分支** | 從最新的 `main` 開分支，例如 `feature/調整結果頁` |
| **用 PR 合併** | 改完後推送分支，到 GitHub 建立 Pull Request |
| **管理員審核** | 僅 **Ohmomomo123** 審核並按 Merge，通過後才會進 `main` |

```
main（正式版，僅審核後更新）
  ↑ Merge（管理員按鈕）
  └── feature/某某功能（你在這裡改）
```

## 一、事前準備

### 1. 安裝 Git

- 下載：<https://git-scm.com/download/win>
- 安裝後確認：

```powershell
git --version
```

### 2. 設定 Git 身分（每台電腦只需做一次）

```powershell
git config --global user.name "你的名字"
git config --global user.email "你的Email@example.com"
```

### 3. 取得 GitHub 存取權限

本儲存庫為 **Private（私人）**，請管理員：

1. GitHub → `Ohmomomo123/skin` → **Settings** → **Collaborators**
2. **Add people** 邀請對方
3. 對方接受邀請後才能 clone / push 分支

### 4. 登入 GitHub（推送時需要）

建議使用 **Personal Access Token（PAT）**：

1. GitHub → **Settings** → **Developer settings** → **Personal access tokens** → **Tokens (classic)**
2. **Generate new token**，勾選 `repo`
3. `git push` 時密碼欄貼上 Token

或使用 [GitHub Desktop](https://desktop.github.com/) 圖形介面登入。

## 三、標準開發流程（分支 → PR）

每次開始新功能或修 bug，請從頭走一遍：

### 步驟 1：更新 main 並開新分支

```powershell
cd C:\路徑\到\skin

# 切到 main 並拉最新
git checkout main
git pull origin main

# 建立功能分支（名稱請用英文或拼音，避免空格）
git checkout -b feature/簡短功能說明
```

**分支命名建議**：

| 前綴 | 用途 | 範例 |
|------|------|------|
| `feature/` | 新功能、介面調整 | `feature/friendly-result-ui` |
| `fix/` | 修 bug | `fix/upload-preview-bug` |
| `docs/` | 只改文件 | `docs/update-readme` |

### 步驟 2：修改並 commit

```powershell
# 改 app.py、templates/index.html 等...

git status
git add .
git commit -m "簡短說明，例如：新增錢癬白話說明"
```

### 步驟 3：推送「你的分支」（不是 main）

```powershell
git push -u origin feature/簡短功能說明
```

第一次推送該分支一定要 `-u`，之後同一分支可只打 `git push`。

### 步驟 4：到 GitHub 建立 Pull Request

1. 開啟 <https://github.com/Ohmomomo123/skin>
2. 通常會出現黃色提示 **Compare & pull request**，點進去  
   或：**Pull requests** → **New pull request**
3. 設定：
   - **base**：`main`（要合進去的目标）
   - **compare**：你的分支，例如 `feature/簡短功能說明`
4. 填寫標題與說明（改了什麼、怎麼測試）
5. 按 **Create pull request**

### 步驟 5：等待管理員審核

- 管理員 **Ohmomomo123** 會檢視程式變更
- 可能留言要求修改 → 你在**同一分支**繼續改、commit、`git push`，PR 會自動更新
- 審核通過後，管理員按 **Merge pull request** 才會進 `main`

### 步驟 6：合併後同步本機

PR 被合併後，你的本機請更新：

```powershell
git checkout main
git pull origin main
git branch -d feature/簡短功能說明
```

## 四、繼續在同一個 PR 上修改

若管理員要求調整：

```powershell
git checkout feature/簡短功能說明
# 修改檔案...
git add .
git commit -m "依審核意見調整說明文字"
git push
```

不必開新 PR，同一個 PR 會自動包含新 commit。

---

## 五、與 main 同步（分支落後時）

若 `main` 已有別人合併的新程式，你的分支可能落後，建議先同步再繼續改：

```powershell
git checkout feature/你的分支
git fetch origin
git merge origin/main
# 若有衝突，解完後：
git add .
git commit -m "合併 main 最新變更"
git push
```

---

## 六、處理衝突（merge conflict）

檔案中若出現：

```
<<<<<<< HEAD
你的內容
=======
別人的內容
>>>>>>> origin/main
```

1. 與管理員或同事溝通要保留哪段
2. 刪除 `<<<<<<<`、`=======`、`>>>>>>>` 標記
3. 存檔後：

```powershell
git add .
git commit -m "解決合併衝突"
git push
```

**不要**對 `main` 使用 `git push --force`，有疑問請聯絡管理員。

## 七、不要上傳到 GitHub 的東西

已由 `.gitignore` 排除，請勿強制加入：

- `__pycache__/`、`*.pyc`、`.venv/`、`venv/`
- `.env`（密碼、API 金鑰）
- `*.log`
- AI 模型權重（執行 `python app.py` 會自動下載到本機快取）

## 九、需要協助時

| 狀況 | 建議 |
|------|------|
| 無法 push 到 main | **正常**，請改 push 你的 `feature/` 分支 |

# 管理員專區（審核者 Ohmomomo123）

以下供儲存庫擁有者設定審核流程與合併 PR。

## A. 建議啟用：main 分支保護

避免協作者直接 push 到 `main`，強制走 PR：

1. 開啟 <https://github.com/Ohmomomo123/skin/settings/branches>
2. **Add branch protection rule**
3. **Branch name pattern**：`main`
4. 建議勾選：
   - **Require a pull request before merging**
   - **Require approvals**：`1`（由你審核）
   - （可選）**Do not allow bypassing the above settings**
   - （可選）**Restrict who can push to matching branches**：只留你自己
5. **Create** / **Save changes**

設定後，協作者只能 push 到 `feature/*` 等分支，合併必須經 PR。

## B. 審核 Pull Request 流程

1. 到 **Pull requests** 分頁
2. 點開待審 PR → 看 **Files changed**
3. 可留言 **Request changes** 或 **Approve**
4. 確認無誤後按 **Merge pull request** → **Confirm merge**
5. 可選 **Delete branch** 清理遠端功能分支

## C. 管理員自己改 main 的方式

你是擁有者，仍建議：

- 小改動：也可 `feature/xxx` → PR → 自己 Merge（保留紀錄）
- 緊急修正：在已設保護規則下，用允許的 bypass 或直接在本機 `main` commit 後 push（若未限制自己）

## D. 邀請新成員

**Settings** → **Collaborators** → **Add people** → 對方接受邀請後，請對方閱讀本文件 **「三、標準開發流程」**。
